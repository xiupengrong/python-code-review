# Python 代码审查详细清单

审查每个维度前，先阅读该维度的检查项，然后使用工具在代码中搜索对应证据。

---

## 1. 架构设计

- [ ] **目录结构**：项目布局是否清晰，符合常见约定（如 `src` 布局、`tests` 目录分离）？
- [ ] **依赖方向**：包与模块间的依赖是否单向、无循环？高层模块是否依赖抽象而非底层细节？
- [ ] **关注点分离**：业务逻辑、数据访问、API 接口、配置管理等是否有效解耦？
- [ ] **单一职责**：模块、类、函数的职责是否明确？是否存在"上帝类"或"万能模块"？
- [ ] **接口与抽象**：是否合理使用抽象基类（`abc`）或协议（`typing.Protocol`），依赖倒置是否到位？
- [ ] **设计模式**：模式使用是否恰当（如工厂、策略、单例），有无过度设计或设计不足？
- [ ] **副作用控制**：模块顶层是否避免了可变全局状态和副作用代码？是否用 `if __name__ == "__main__"` 保护入口？
- [ ] **配置管理**：配置是否与代码分离？多环境（开发/测试/生产）配置是否易于切换且安全？

**搜索提示：** 查找模块级赋值（全局状态）、`import` 语句（循环依赖）、超大文件（>500行）、超大类（>200行）

## 2. 安全性

- [ ] **输入验证**：所有外部输入（用户、文件、网络）是否进行了类型、范围、格式验证与净化？
- [ ] **SQL 注入防护**：数据库操作是否使用参数化查询或 ORM，严禁拼接 SQL 字符串？
- [ ] **命令注入防护**：是否避免 `os.system` 及 `shell=True`？使用 `subprocess` 时参数是否列表化且不引入用户输入？
- [ ] **反序列化安全**：是否禁止用 `pickle` 加载不可信数据？优先使用 `json` 或安全序列化格式。
- [ ] **敏感信息处理**：密钥、密码、Token 是否存储在环境变量或密钥管理服务中？绝无硬编码或误提交。
- [ ] **路径遍历防护**：文件路径拼接时是否校验基础路径，使用 `os.path.realpath` 等防止目录穿越？
- [ ] **依赖漏洞**：是否定期审计第三方依赖漏洞（工具：`pip-audit`、`safety`）？依赖版本是否保持更新？
- [ ] **日志安全**：日志中是否避免了记录敏感信息（密码、令牌、个人身份信息）？
- [ ] **认证授权**：认证机制是否健壮，授权检查是否覆盖所有受保护端点？是否存在权限绕过风险？
- [ ] **加密实践**：是否使用现代算法（bcrypt/scrypt/argon2 哈希密码，AES-GCM 加密），杜绝 MD5/SHA1 用于安全目的？
- [ ] **动态执行**：是否完全避免 `eval`、`exec`、`compile` 等动态执行不可信代码？
- [ ] **资源与超时**：网络请求、文件操作等是否设置了合理超时和大小限制，防止资源耗尽（DoS）？

#### 金融安全规范（当项目涉及金融业务时，以下检查项必须执行）

> **激活条件：** 项目中存在支付、转账、账户管理、清结算、交易记录、金额计算等金融业务代码。
> 判断依据见 SKILL.md "识别项目领域" 步骤。

- [ ] **金额计算精度**：是否使用 `decimal.Decimal` 进行金额计算和存储？是否杜绝 `float` 用于金额运算？是否正确设置和统一精度（`decimal.getcontext().prec`）？是否采用正确的舍入规则（如 `ROUND_HALF_UP`、`ROUND_BANKERS`）？金额比较是否使用 `Decimal.quantize()` 而非 `==`？
- [ ] **事务完整性**：金融交易是否保证原子性（ACID）？是否实现幂等性机制（幂等键、去重表）防止重复提交？异常时是否有可靠的回滚或冲正逻辑？分布式场景是否采用 Saga/TCC 等一致性方案？是否有防止重复扣款的机制？
- [ ] **金融数据保护**：银行卡号、账户号码是否脱敏存储（仅存后四位或令牌化）？传输是否使用 TLS 1.2+？是否实现字段级加密（对敏感金融字段单独加密，而非仅传输层加密）？是否遵循 PCI DSS 要求（不存储 CVV、卡号脱敏等）？PII（个人身份信息）是否加密存储和传输？
- [ ] **审计追踪**：所有金融操作是否记录不可变审计日志（append-only，禁止修改/删除）？日志是否包含完整上下文（操作人、时间戳、金额、操作类型、前后状态、请求 ID）？是否实现操作溯源（可还原完整交易链路）？关键操作是否有独立审计表？
- [ ] **反欺诈与风控**：是否实现交易频率限制（velocity check）和金额阈值控制？是否有风控规则引擎（规则匹配、异常检测）？异常交易是否触发熔断/拦截机制？是否有可疑交易告警和自动拦截？大额操作是否有额外验证（如二次确认、短信验证）？
- [ ] **支付安全**：支付令牌化处理是否符合行业标准（如 PCI DSS tokenization）？支付通道是否全程加密（TLS 1.2+，证书锁定）？是否实现完整的支付状态机（含明确的状态转移规则，杜绝非法状态跳转如 `refunded → paid`）？第三方支付回调是否验证签名（HMAC/RSA）并做幂等处理？是否有超时关单机制？
- [ ] **对账与职责分离**：是否实现对账逻辑（交易流水与银行/渠道流水定期核对）？差异账是否有告警和人工处理流程？资金相关操作是否实现双人操作/多级审批（maker-checker 模式）？关键配置变更是否有审批工作流？
- [ ] **合规要求**：是否根据业务需要实现 KYC（身份验证）流程？是否有反洗钱（AML）相关监控规则（大额交易、可疑模式）？金融数据保留期限是否符合监管要求（如交易记录保存 5 年以上）？是否支持监管数据导出和报送？敏感数据销毁是否有安全擦除机制？

**搜索提示：** `grep` 以下模式：
- `os.system(`, `shell=True`, `subprocess.call(shell=`
- `eval(`, `exec(`, `compile(`
- `pickle.load(`, `pickle.loads(`
- `password =`, `secret =`, `token =`, `api_key =`（硬编码敏感信息）
- `md5(`, `sha1(`（弱哈希）
- 字符串拼接 SQL：`f"SELECT`, `"SELECT %s" %`, `"SELECT" +`
- `requests.get(` 或 `requests.post(` 不带 `timeout=`

**金融安全搜索提示：** 在通用安全搜索基础上，额外搜索：
- `float` 与金额变量同时出现：`float.*amount`, `float.*price`, `float.*balance`, `float.*fee`
- 缺少 Decimal 的金额计算：搜索 `amount`, `price`, `total`, `balance` 等变量但文件未 `import Decimal`
- 支付相关：`payment`, `pay(`, `transfer(`, `refund(`, `withdraw(`, `charge(`
- 交易关键词：`transaction`, `order`, `settle`, `reconcil`
- 银行卡/卡号：`card_number`, `bank_card`, `cvv`, `pan`（不应以明文出现在代码或日志中）
- 审计日志：`audit`, `trail`, `log.*transaction`（确认存在性）
- 幂等性：`idempoten`, `dedup`, `unique.*key`（确认金融操作有幂等保障）
- 状态机：`status`, `state` 与 `payment`/`order` 同时出现（检查状态转移逻辑）

## 3. 可读性

- [ ] **PEP 8 一致性**：代码风格是否遵循 PEP 8（可用 `flake8`、`black` 检查）？团队是否有统一格式化配置？
- [ ] **命名规范**：变量/函数使用 `snake_case`，类用 `PascalCase`，常量用 `UPPER_CASE`；名称是否达意且不误导？
- [ ] **函数复杂度**：函数/方法是否短小（建议 <30 行），参数数量是否合理（建议 ≤5）？复杂逻辑是否已抽离？
- [ ] **类型注解**：是否合理使用类型提示，提高可读性并便于静态检查（配合 `mypy`）？
- [ ] **注释质量**：注释是否解释"为何如此"而非"做了什么"？是否避免了冗余或过时的注释？
- [ ] **文档字符串**：公开模块、类、函数是否包含清晰 docstring，并遵循统一风格（如 Google、NumPy 风格）？
- [ ] **魔法数字/字符串**：是否用命名常量替代硬编码的魔法值？
- [ ] **导入规范**：导入是否分组（标准库 → 第三方 → 本地）并按字典序排列？是否避免 `import *`？
- [ ] **现代 Python 特性**：是否善用 f-string、海象运算符（`:=`）、数据类、路径库（`pathlib`）等提升表达力？
- [ ] **空白与布局**：合理的空行与缩进，逻辑块之间是否有视觉分隔，提升扫描效率？

**搜索提示：** 查找超长函数、`import *`、单字母变量名（循环变量除外）、缺少 docstring 的公开函数/类

## 4. 健壮性

- [ ] **精准异常处理**：是否捕获特定异常，避免宽泛的 `except Exception: pass`？是否在合适层级处理并记录异常？
- [ ] **资源管理**：文件、网络连接、锁等是否使用 `with` 语句或 `try-finally` 确保释放？
- [ ] **重试与退避**：对外部服务的瞬态故障是否实现了重试机制（指数退避、最大重试次数）？
- [ ] **边界条件**：空容器、`None`、零值、负数、超长字符串等边缘情况是否测试并处理？
- [ ] **并发安全**：多线程/异步下共享状态是否使用锁或线程安全结构？是否避免死锁风险？
- [ ] **可变默认参数**：函数签名中是否避免了可变对象作默认参数（如 `def f(l=[])`）？
- [ ] **日志与监控**：是否记录了关键操作、异常和性能指标，日志级别是否合理？
- [ ] **输入防御性检查**：函数入口是否通过断言或显式检查拒绝非法参数（如类型、范围）？
- [ ] **编码处理**：文件读写、网络 IO 是否明确指定编码（通常 `utf-8`），避免平台差异？
- [ ] **降级与容错**：非核心功能故障时是否可优雅降级，不影响主流程（例如缓存不可用时回源）？

**搜索提示：** 查找以下模式：
- `except:` 或 `except Exception:` 不带 `raise` 或日志
- `except: pass`（吞掉异常）
- `open(` 不在 `with` 语句中
- `def f(x=[]` 或 `def f(x={}`（可变默认参数）
- `open(` 不带 `encoding=` 参数

## 5. 可扩展性

- [ ] **开闭原则**：新增功能是否通过扩展（新模块/类）而非修改已有代码实现？
- [ ] **依赖注入**：关键依赖是否通过参数/构造函数注入，方便替换实现与测试？
- [ ] **策略与插件化**：是否通过策略模式、注册机制、入口点（`entry_points`）支持动态扩展？
- [ ] **配置驱动**：行为变更是否可通过配置文件而非代码改动实现？配置格式是否一致？
- [ ] **API 兼容性**：对外接口是否考虑版本化？是否避免破坏性变更或提供过渡期？
- [ ] **数据模型扩展**：数据库表/文档模型是否预留灵活性（如 JSON 字段、EAV 模式适度使用）？
- [ ] **硬编码消除**：业务规则、阈值、路径等是否抽离为配置或常量，便于集中调整？
- [ ] **组合优于继承**：在行为复用上是否优先使用组合/委托，避免过深的继承层次？
- [ ] **条件逻辑简化**：冗长的 `if-elif` 链是否用映射表、策略分派等方式替代？

**搜索提示：** 查找超长 `if-elif` 链、深层继承（>2层）、硬编码的 URL/路径/阈值、直接实例化具体类（而非通过接口）

## 6. 性能与效率

- [ ] **循环优化**：循环内是否避免重复计算和不必要的对象创建？是否使用列表推导式/生成器表达式替代低效循环？嵌套循环中的不变量是否已外提？
- [ ] **算法与数据结构**：算法复杂度是否合理？成员检测是否使用 `set`/`dict`（O(1)）而非 `list`（O(n)）？排序是否选用合适算法？是否有不必要的 O(n²) 操作？
- [ ] **异步与多进程**：I/O 密集型任务是否使用 `asyncio` 或线程池？CPU 密集型任务是否使用 `multiprocessing`？混用场景是否正确处理事件循环阻塞？
- [ ] **数据库优化**：查询是否避免 N+1 问题？是否使用连接池？是否有缺失的索引（ORM 中检查 `index=True`）？批量操作是否使用 `bulk_create`/`executemany`？
- [ ] **缓存策略**：是否使用缓存（`functools.lru_cache`、Redis、Memcached）减少重复计算？缓存过期策略是否合理？是否有缓存穿透/雪崩/击穿风险？
- [ ] **内存管理**：是否注意大列表的内存占用？是否使用生成器（`yield`）和惰性求值处理大数据集？是否有内存泄漏隐患（如全局缓存无限增长）？
- [ ] **生成器与惰性求值**：处理大数据集或流式数据时是否使用生成器而非一次性加载到内存？文件读取是否逐行处理（`for line in file`）而非 `file.read()`？
- [ ] **IO 与连接池**：HTTP 客户端是否使用连接池（`requests.Session`、`httpx.Client`）？数据库/Redis 连接是否池化？文件 IO 是否使用缓冲？

**搜索提示：** 查找循环内的数据库查询（N+1）、`list` 用于查重（应用 `set`）、缺少 `@lru_cache` 的纯函数重复调用、嵌套循环中的不变量计算、`float` 用于金额（与金融安全交叉检）

## 7. 测试与质量保障

- [ ] **测试覆盖率**：核心业务逻辑是否有单元测试覆盖？关键路径（如支付流程、权限校验）是否有集成测试？覆盖率是否达到合理阈值（建议核心模块 ≥80%）？
- [ ] **边界与异常测试**：异常路径、边界条件（空值、极值、并发竞争）是否包含在测试用例中？是否有故障注入测试？
- [ ] **测试独立性**：测试是否可独立运行，不依赖特定顺序或外部状态？测试间是否做到完全隔离？
- [ ] **Mock 与 Fixture**：外部依赖（数据库、API、文件系统）是否使用 mock/fixture 隔离？金融相关测试是否使用精确的 Decimal mock 数据？
- [ ] **CI/CD 集成**：是否配置了 CI 流水线自动运行 lint、类型检查、测试和安全扫描？是否在合并前强制通过所有检查？
- [ ] **测试质量**：测试用例是否有意义（不是为覆盖率凑数）？断言是否具体明确？是否避免了测试间的脆弱依赖？
- [ ] **性能与变异测试**（可选）：是否有性能基准测试（`pytest-benchmark`）？是否考虑变异测试（`mutmut`）验证测试用例有效性？

**搜索提示：** 检查 `tests/` 目录存在性和测试文件数量、CI 配置文件（`.github/workflows/`、`.gitlab-ci.yml`、`Jenkinsfile`）、测试中是否使用了 `fixtures`/`mock`/`patch`、`pytest.ini`/`conftest.py` 存在性、`.coveragerc` 或 `pyproject.toml` 中的覆盖率配置

## 8. 依赖与部署

- [ ] **依赖必要性与锁定**：直接依赖是否都是必要的？是否避免了版本冲突？锁文件（`requirements.txt` 含确切版本、`Pipfile.lock` 或 `poetry.lock`）是否提交？
- [ ] **环境隔离**：是否使用了虚拟环境（`venv`、`conda`、`poetry`）进行隔离？是否有 `.python-version` 或运行时版本约束？
- [ ] **容器化**：Docker 镜像构建是否精简、安全？是否使用多阶段构建？是否避免以 root 运行？是否有 `.dockerignore`？
- [ ] **安全审计**：是否定期审计依赖漏洞（`pip-audit`、`safety check`）？CI 中是否集成依赖安全扫描？
- [ ] **依赖分离**：是否分离生产依赖和开发依赖（`requirements-dev.txt`、`[tool.poetry.group.dev.dependencies]`）？
- [ ] **Python 版本兼容**：是否明确支持的 Python 版本范围？`pyproject.toml` 中是否声明 `python_requires`？CI 是否进行多版本矩阵测试？

**搜索提示：** 检查 `requirements.txt`/`pyproject.toml`/`Pipfile`、`Dockerfile`（是否用了 slim/alpine 基础镜像、是否以 root 运行、多阶段构建）、`.dockerignore` 存在性、`tox.ini`（多版本测试）、`.python-version` 文件

## 9. 文档记录

- [ ] **README 完整性**：项目 README 是否包含概述、安装、运行、测试说明？是否有快速开始（Quick Start）示例？
- [ ] **设计决策记录**：重大设计决策是否有记录（ADRs）或代码注释说明？复杂算法和非显而易见的逻辑是否有解释？
- [ ] **API 文档**：API 变更是否同步更新文档（如 OpenAPI/Swagger）？对外暴露的接口是否有完整的参数说明和示例？
- [ ] **变更日志**：是否有 CHANGELOG 记录版本变更？是否遵循语义化版本（SemVer）？
- [ ] **运维文档**（如适用）：是否有部署指南、故障排查手册、监控告警说明？

**搜索提示：** 检查 `README.md` 存在性和内容完整性、`docs/` 目录、ADR 文件（`docs/adr/`）、API schema 文件、`CHANGELOG.md`、`CONTRIBUTING.md`
